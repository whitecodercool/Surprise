🎯 What Problem Does Ghost Stack Solve?

Imagine you're at a school or office, and someone has hired a security guard (the firewall) to stand at the front door. This guard has a list of websites you're not allowed to visit — YouTube, Twitter, Reddit, etc.

When you try to visit one of those websites, the guard sees the name on your "letter" (your internet request) and says "Nope! Blocked!"

💡 In Simple Words

Ghost Stack is like a clever postal system that lets your letters reach their destination without the security guard ever seeing which website you're writing to. It doesn't use VPNs, doesn't use Tor, and doesn't send your traffic through another country. It simply changes how your computer talks so the guard can't understand what's being said.

📖 Part 3: The Secret Phone Book — DNS over HTTPS

📁 src/ghoststack/dns/DoHClient.ts

Every time you visit a website, your computer first looks up the website's "real address" (IP address) in a phone book (DNS). The problem? The network controls that phone book and can lie to you or refuse to give you the address.

Ghost Stack has its own secret, encrypted phone books that the network cannot read or tamper with:

ProviderAddressType
📗 Cloudflare1.1.1.1Primary
📘 Google8.8.8.8Secondary
📙 NextDNSdns.nextdns.ioSecondary
📕 Quad9dns.quad9.netSecondary
⚡

The Clever Part: Ghost Stack asks all 4 phone books at the same time and picks the answer that at least 2 agree on. This way, even if one phone book is compromised, the majority still gives the correct answer. This is called "consensus resolution."

🌍 Real-World Analogy

Instead of trusting one person for directions, you ask 4 different strangers. If 3 of them say "turn left," you turn left — even if one person tried to send you the wrong way.

🗺️ Part 4: The Address Finder — Live IP Discovery

📁 src/ghoststack/core/LiveIPDiscovery.ts

Once the Doctor diagnoses that a website is blocked, the Address Finder's job is to discover the website's real home address. It uses 4 methods, tried one after another:

1

Method 1: Ask the Secret Phone Books (DoH Consensus)

Ask all 4 encrypted phone books and find the address they agree on. This is the fastest method and works in most cases.

2

Method 2: CDN Neighborhood Scan

If the website lives in a well-known "neighborhood" (like Cloudflare or Google's servers), Ghost Stack generates random addresses in that neighborhood and knocks on doors to see which one has a valid key (certificate) for the website.

🌍 Analogy

Like checking every house on a street to find which one belongs to your friend.

3

Method 3: Certificate Transparency Logs

Public records exist of every security certificate ever issued to a website. Ghost Stack queries these public records (via crt.sh) to find the website's associated domains and then resolves them.

🌍 Analogy

Like checking public property records to find someone's address.

4

Method 4: TemporalDNS (Last Resort)

The slowest but most creative method — uses timing-based encoding over DNS queries to extract the real IP even when all other methods are blocked.

🌍 Analogy

Like communicating in Morse code when all other communication is blocked.

🎭 Part 5: Engine #1 — IPRaw (The Disguised Letter)

📁 src/ghoststack/ipraw/IPRawEngine.ts

Now that we know the website's real address, IPRaw tries to connect to it while hiding the website name from the guard. It tries 3 tricks in order:

🚇

Trick A: QUIC — The Fast Underground Tunnel

Uses a newer, faster protocol (like a different type of road) that many firewalls don't know how to inspect. Think of it as taking an underground tunnel instead of the main road.

✉️

Trick B: ECH — Encrypted Client Hello

Normally when you connect to a website, you have to show the website name on the outside of your "envelope" (in plaintext). ECH encrypts the website name so the guard only sees something like "cloudflare.com" on the outside — which is allowed — while the real destination is hidden inside.

🌍 Analogy

Like writing "To: Google" on the outside of an envelope, but the real letter inside says "To: YouTube."

🔐

Trick C: Direct TLS 1.3 Connection

A straightforward encrypted connection directly to the website's IP address, using the strongest available encryption (TLS 1.3). Like calling someone's phone directly instead of going through the switchboard.

✂️ Part 6: Engine #2 — SplitCast (The Shredded Envelope)

📁 src/ghoststack/splitcast/SplitCastEngine.ts

If IPRaw's tricks don't work, SplitCast takes a completely different approach. Here's the key insight:

💡

The firewall guard reads the website name from the very first "hello" message your computer sends (called the TLS Client Hello). If the guard can't read that message, it can't block you!

SplitCast tears up that "hello" message into tiny pieces and sends them one at a time, with small random pauses in between (10–60 milliseconds). Each piece is too small for the guard to understand, and the guard can't reassemble them fast enough.

How It Looks

// ❌ Normal connection (guard can read the website name) Your Computer ──[ "Hi, I want to visit YouTube.com" ]──→ Guard sees "YouTube" → BLOCKED! // ✅ SplitCast connection (guard sees only meaningless fragments) Your Computer ──[ "Hi," ]──→ Guard: ??? Your Computer ──[ " I w" ]──→ Guard: still confused... Your Computer ──[ "ant " ]──→ Guard: can't read it... Your Computer ──[ "YouT" ]──→ Guard: gives up, lets it pass ✓ Your Computer ──[ "ube." ]──→ Server: collects all pieces ✅

Why does the real server still work? Because the server is designed to wait and collect all the pieces before reading the message. The firewall guard is not — it needs to make a split-second decision.

SplitCast can split the message into 3, 5, or 7 fragments. If one setting doesn't work, it automatically tries another.

🌍 Real-World Analogy

Imagine a security guard checks every letter for banned words. If you tear your letter into tiny strips and mail them separately over 7 envelopes, the guard can't reassemble them — but your friend at the destination can tape them back together.

📬 Part 7: The Invisible Mailroom — Local Proxy

📁 src/ghoststack/core/GhostStackProxy.ts

This is the central piece that ties everything together. Think of it as a tiny, invisible mailroom sitting inside your own computer (at address 127.0.0.1 — which means "myself").

How the Mailroom Works

Step 1

All mail goes through the mailroom

Ghost Browser tells the built-in Chromium engine: "Send ALL your traffic through our internal mailroom." Every single web request passes through it.

Step 2

Look up the real address

The mailroom checks its own encrypted phone books (DoH) for the website's real IP address — ignoring the network's potentially fake phone book.

Step 3

Connect directly to the real IP

Opens a TCP connection straight to the website's real server, bypassing any DNS-level blocks.

Step 4 — The Magic

1-byte TLS split on EVERY connection

Sends just the first byte of the "hello" message, waits 50 milliseconds, then sends the rest. This tiny split prevents the firewall from ever reading the website name.

Step 5

Transparent data flow

After the initial handshake, data flows normally between the browser and the website. The user sees nothing unusual.

⚡

Important: The 1-byte TLS split happens on EVERY connection, not just blocked ones. This means the firewall never gets a chance to see which websites you're visiting — even for sites that aren't on the block list.
